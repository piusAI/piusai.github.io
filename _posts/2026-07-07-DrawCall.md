##  DrawCall?

DrawCall은 CPU가 GPU한테 **정보로 뭔가를 그러라** 며 보내는 명령이다

>이 **Mesh (vtx data)** 를 Shader / Texture / Material 을 써서 이 위치에 그려라

이게 한번의 Draw Call
- DX :  `DrawIndexed()`, `Draw()`  DX API 호출
- Unity : `Graphics.DrawMesh()` -> 내부적으로 draw call 하나 발생

Draw call 자체는 **그려라** 는 명령일 뿐이고, 이게 있어야 화면에 뭔가 나온다.

: 최종적으로 화면에 나오는 것은 항상 메모리 버퍼에 담는 저장 창고가 된다.

Draw Call 명령을 통해 Render Pipeline에서 구동되면 최종적으로 **OM**에서 `depthstencil` 로 pixel 깊이/색상 연산 마친뒤 지정된 RenderTarget( Back buffer / RenderTexture)메모리에 최종 기록된다!
Draw call은 최종 버퍼를 채우기 위한 Trigger!!


---

### Draw Call FLOW

DrawCall하나를 CPU -> GPU에 보내려면 그 전에 다음 과정이 필요하다

1. 사용할 Shader GPU에 바인딩
2. Texture/material 쓸지 세팅  ***Render state change*** - 병목시 주된 범인!
3. Vertex/Index Buffer를 쓸지 세팅
4. Graphics Driver를 거쳐서 명령을 GPU 커멘드 Queue에 밀어넣기

이 준비과정을 매 **drawcall 마다** 반복해야하는데,

- Object 한개일때는 문제가 없음
- Scene상에 object 5000개 있고 material마다 제각각이라면?
5000번의 Drawcall : CPU가 위 FLOW작업을 5000번 반복 

> CPU가 GPU한테 일 시키는것 자체의 **병목** !


### 최적화 기법
이로부터 draw call 수를 줄이기위한 최적화 기법들이 필요해진다.
- **Instancing** — 같은 mesh를 여러 번 그릴 때 하나의 draw call로 처리
- **Texture Atlas** — 여러 텍스처를 하나로 합쳐서 material 전환 횟수 줄이기
- **Batching** — 여러 object를 하나의 draw call로 합치기 (Static/Dynamic Batching, SRP Batcher 등)



### 정리
> DrawCall의 병목은 GPU를 느리게 만드는것 자체가 아니라 CPU에 overhead가 쌓여서 전체 프레임이 느려지는 것!

- Draw Call에서의 병목은 State Change가 진짜 주범!
Render State Change 오버헤드라고 부르고, CPU가 driver를 거쳐 GPU 명령큐(Command Queue)에 명령을 쌓는 과정에서 CPU코어 하나에 심각한 병목 !

RenderThread 병목을 유발한다!
